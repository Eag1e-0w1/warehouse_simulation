
### Задача 
Разработка прототипа упрощенного аналитического сервиса для расчёта и сравнения эффективности складского 
процесса до и после внедрения мобильных роботов (AMR). 

### Допущения в модели

1.  В исходный файл excel добавлен показатель "Среднее реальное время работы комплектовщика"
2.  В исходный файл excel добавлены показатели параметров роботизации для дальнейшего расширения функционала модели.
3.  В исходный файл excel добавлен коэффициент "Продуктивность". В данной модели количество роботов не релевантно. Расчет отталкивается от коэффициента продуктивности. Для сценария "Человек-к-товару" коэффициент принимается как 1.0, при сценарии "Товар-к-человеку" - 1.4.
4.  В файл **config.py** вынесены переменные, не имеющие отношения к непосредственно процессу моделирования: PATH (путь к исходному файлу excel), SIMULATION_DAYS (горизонт планирования) и PEAK_RATIO (прирост нагрузки на комплектовщиков в пиковый период)

### Выводы

В ходе моделирования складского процесса подтверждена эффективность перехода на схему «Товар-к-человеку».  
При внедрении роботов и повышении коэффициента продуктивности (`productivity > 1.0`) система демонстрирует рост пропускной способности и сокращение среднего времени отбора одного пика.  
Один из основных эффектов — сглаживание пиковых нагрузок в интервале 20:00–05:00.  
Финансовый расчёт показывает, что капитальные затраты (CAPEX) на роботов способны окупиться за счет уменьшения операционных затрат (ОРЕХ) на определенном временном горизонте.

### Направления развития сервиса включают:

1.  **Расширение модели:** переход от детерминированного распределения спроса к вероятностному (например, распределение Пуассона) для более реалистичного моделирования «шума».
    
2.  **AMR:** учёт времени подзарядки, скорости перемещения для более точного расчёта эффекта роботизации.
    
3.  **Интеграция с внешними системами:** добавление модулей для загрузки реальных исторических данных и расчёта NPV с учётом инфляционных ожиданий и изменяющейся ключевой ставки.
    
4.  **Взаимодействие с пользователем:**  
    На текущем этапе система работает в формате **CLI-утилиты**. Пользователь задаёт входные параметры через редактирование исходного файла конфигурации и входного Excel-файла. После запуска скрипта система выполняет цикл расчёта и выводит итоговый набор метрик в консоль. Модули (`Simulator`, `Calculator`) готовы к подключению веб-интерфейса (например, Streamlit) или API без изменения ядра бизнес-логики.
    
    ### 2\. Архитектурное описание решения
    

Решение построено по модульному принципу с разделением ответственности между слоями данных, бизнес-логики и расчётов. Взаимодействие компонентов осуществляется последовательно: от валидации входных данных к имитации процесса и финальной агрегации метрик.

**Организация логики и данных**

| Компонент | Ответственность | Ключевые технологии |
| --- | --- | --- |
| **Data Layer** (`models.py`) | Описание схемы данных, валидация типов, бизнес-правила (`model_validator`), преобразование форматов (`computed_field`). | `Pydantic v2`, `pandas` |
| **Simulation Core** (`simulator.py`) | Детерминированная пошаговая имитация (1 шаг = 1 час). Расчёт баланса «Спрос → Очередь → Выполнение» с учётом пикового профиля и доступной мощности. | `pandas`, `math` |
| **Analytics Layer** (`calculator.py`) | Агрегация временных рядов в бизнес-метрики. Расчёт экономики (ФОТ, амортизация), операционных показателей (задержки, время пика) и финансовой эффективности. | `statistics` |
| **Configuration** (`config.py`, `apis.py`) | Управление глобальными константами симуляции и получение внешних макроэкономических данных (ключевая ставка). | `requests` |

**Структура проекта**  
warehouse_simulation/  
├── main.py # Точка входа (CLI)  
├── models.py # Pydantic-схемы: валидация, алиасы, computed_field  
├── simulator.py # Ядро имитации: почасовой цикл, очереди, capacity  
├── calculator.py # Агрегация метрик: OPEX, CAPEX, KPI, финансы  
├── apis.py # Внешние API: загрузка ставки ЦБ  
├── config.py # Глобальные константы (длительность симуляции, пики)  
├── requirements.txt # Фиксированные зависимости  
├── dump/  
│ └── data.xlsx # Исходные данные из ТЗ  
└── README.md # Документация


**Архитектурные принципы**

| Принцип | Реализация в проекте |
| --- | --- |
| **Разделение ответственности** | `Simulator` генерирует данные, `Calculator` считает метрики, `models` валидирует вход |
| **Валидация на границе** | Все внешние данные проходят через Pydantic до попадания в бизнес-логику |
| **Детерминированность** | Симуляция воспроизводима: одинаковые входные → одинаковый результат |
| **Расширяемость** | Добавление нового сценария = `.model_copy(update={...})`, новый KPI = метод в `Calculator` |
| **Независимость от UI** | Ядро не привязано к интерфейсу: можно запускать через CLI, тесты или будущий Streamlit/FastAPI |
